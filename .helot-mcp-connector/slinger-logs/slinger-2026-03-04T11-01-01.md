**Code Duplication Patterns Found:**

1.  **Connection Open/Close Pattern:**
    *   **Location:** `data/solvency.py`, `data/assets.py`, `data/profitability.py` (and others).
    *   **Pattern:** Repeated `sqlite3.connect(DB_NAME)` followed by `conn.close()`.
    *   **Evidence:**
        *   `solvency.py`: `conn = sqlite3.connect(DB_NAME)` (line 13) -> `conn.close()` (line 19).
        *   `solvency.py`: `conn = sqlite3.connect(DB_NAME)` (line 76) -> `conn.close()` (line 196).
        *   `assets.py`: `conn = sqlite3.connect(DB_NAME)` (line 13) -> `conn.close()` (line 16).
    *   **Risk:** Inconsistent error handling (some use `try/except` blocks to close, others do not).

2.  **SQL Query Construction:**
    *   **Location:** `data/solvency.py`, `data/assets.py`.
    *   **Pattern:** Repeated use of `to_sql_list()` and string formatting for `IN` clauses.
    *   **Evidence:**
        *   `solvency.py`: `leis_str = to_sql_list(lei_list)` used in multiple queries.
        *   `assets.py`: `leis_str = to_sql_list(lei_list)` used in `get_assets_kpis`.
        *   **Duplication:** The query structure `SELECT f.lei, ... FROM facts_oth f JOIN institutions i ON f.lei = i.lei WHERE f.lei IN ({leis_str}) ...` is repeated with minor variations across `solvency.py` and `assets.py`.

3.  **Weighted Average Logic:**
    *   **Location:** `data/solvency.py`, `data/assets.py`, `data/profitability.py`.
    *   **Pattern:** Repeated `group_weighted_avg` or inline `weighted_avg` calculations.
    *   **Evidence:**
        *   `solvency.py`: `group_weighted_avg(group, 'CET1 Ratio', 'TREA')` (line 125).
        *   `assets.py`: Inline `weighted_avg` function defined inside `get_assets_averages` (line 53).
        *   **Duplication:** The logic for calculating weighted averages based on TREA or Total Assets is duplicated rather than centralized in `data_utils.py`.

4.  **Safe Ratio/Division:**
    *   **Location:** `data/solvency.py`, `data/assets.py`.
    *   **Pattern:** Repeated `safe_ratio` or `lambda x: x / y if y > 0 else 0` patterns.
    *   **Evidence:**
        *   `solvency.py`: `safe_ratio(x['AT1 Capital'], x['TREA'])` (line 42).
        *   `assets.py`: `lambda x: x['Loans and advances'] / x['Total Assets'] if x['Total Assets'] > 0 else 0` (line 27).

**Conclusion:**
The largest files are `data/benchmarking.py` (858 lines) and `ui/tabs/benchmarking_dashboard.py` (418 lines). Significant code duplication exists in database connection handling, SQL query construction, and financial ratio calculations across the `data/` module. Refactoring these into shared utility functions in `data_utils.py` would reduce monolithic size and improve maintainability.
